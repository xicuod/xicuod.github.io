## SQL vs. NoSQL

SQL 是关系型数据库的查询语言，NoSQL 是非关系型数据库的查询语言。

### 数据结构：结构化的 vs. 非结构化的

SQL 是结构化的 (structured)，有表，表中有字段，可以给字段加各种约束：主键约束、唯一约束、无符号约束（不能是负数）；可以给字段定义数据类型和数据长度，比如 `id` 是 `bigint(20)`，`name` 是 `varchar(32)`，`age` 是 `int(3)`。而这些约束一旦定义好，表的结构就定义好了，插入的数据要严格遵守这些约束，否则就会报错。不建议之后再随便修改表的结构，这时表的中的数据可能上千万行，如果修改某个字段，会影响大规模的数据。而且，表的结构跟业务也息息相关，如果表的结构变了，那么业务逻辑可能也要跟着变。总之，关系型数据库是不够灵活的，适合需求稳定不多变的业务场景。

NoSQL 是非结构化的，常见的 NoSQL 数据库有四种：键值类型、文档类型、列类型和图类型。

- 键值类型数据库是按一个个键值对 (key-value pair) 存储的，键和值都可以是任何数据类型，如 Redis。
- 文档类型数据库是按一条条文档 (document) 存储的，每条文档保存一个实体的 JSON 字符串，如 MongoDB。
- 列类型数据库就是把一条记录当作一个列存储，查询的时候也按列查，如 HBase。
- 图类型数据库就是存的每条数据都是一个节点，节点和节点之间可以有联系，于是可以用线把它们连在一起，这样节点和节点之间的线就构成了一张图 (graph)，如 Neo4j。

NoSQL 对数据的结构要求很宽松，多一条或少一条字段都没有太大影响。

### 数据关联：关联的 vs. 无关联的

SQL 是关联的 (relational)，关系型数据库用外键来建立实体与实体之间的关系，外键关系一旦建立，就不能随便删除数据了，删除一条数据可能牵动几十上百、成百上千、成千上万条数据。

NoSQL 是无关联的，它维护实体之间的关系是用 JSON 的方式维护的，比如给用户的 JSON 里面嵌套一个订单数组字段：

```json
{
    "id": 1,
    "name": "张三",
    "orders": [
        {
            "id": 1,
            "item": {
                "id": 10, "title": "手机A", "price": 4999
            }
        },
        {
            "id": 2,
            "item": {
                "id": 20, "title": "手机B", "price": 3999
            }
        }
    ]
}
```

当然，李四也可能买手机 A 或手机 B，这样就会把手机 A 和手机 B 的数据重复存储，造成冗余。但如果只存手机 A 和手机 B 的 `id` 的话，就需要开发者在业务逻辑层维护这个关联。

### 查询方式：SQL 查询 vs. 非 SQL 查询

SQL 查询的格式和语法是固定的，只要是关系型数据库都能用 SQL 查询。

NoSQL 的查询没有固定的语法格式，在 Redis 里面，查询语句是这样的：

```
get user:1
```

其中 `user:1` 是一个字符串，指的就是 `id` 为 `1` 的用户。

而 MongoDB 的查询语句是函数调用的形式：

```
db.users.find({_id: 1})
```

而 Elasticsearch 的查询语句是 HTTP 请求的形式：

```
GET http://localhost:9200/users/1
```

可以看到 NoSQL 的语法虽然千奇百怪，但是都相对简单，不像 SQL 那样语法森严。然而，缺点就在于不同的 NoSQL 数据库的查询语法都不一样，每次都要新学。

### 事务特性：ACID vs. BASE

事务必须满足 ACID 的特性，也就是原子性、一致性、隔离性和持久性。

对于 SQL，所有的关系型数据库都满足事务的 ACID 特性。对于 NoSQL，这些数据库要么压根没有事务，要么只能做到一些基本的一致性，甚至无法保证一致性，因此称为 BASE，表示事务的基本可用、软状态和最终一致性。

### 其他差异

- 存储方式：SQL 磁盘，NoSQL 内存
- 扩展性：SQL 垂直（数据都存在本机，分布式支持较差，只能靠升级本机配置提升性能），NoSQL 水平（分布式支持好，基于数据 `id` 做哈希运算，根据不同的哈希值存储到不同的节点，天然支持数据拆分）
- 使用场景：SQL 适合数据结构固定，业务对数据安全性和一致性要求较高的场景；NoSQL 适合数据结构不固定，业务对数据安全性和一致性要求不高，且对性能要求较高的场景。
