异常就是程序出现的问题。不是让我们以后不出异常，而是程序出了异常之后，该如何处理。

实际工作中，遇到的情况不可能是非常完美的：

- 你写的某个模块，用户的输入不一定符合你的预期
- 你的程序要打开某个文件，这个文件可能不存在或者文件格式不对
- 你要读取数据库的数据，数据可能是空的
- 程序在跑着的，内存或硬盘可能满了

软件程序在运行过程中，非常可能遇到这些例外情况。对于这些例外情况，怎么让程序做出合理的处理，而不至于程序崩溃，就是程序员处理异常所要做的事。

- 异常是程序运行中出现的各种“欧亨利式”（意料之外、情理之中）的状况：文件找不到、网络连接失败、非法参数等。
- 异常发生在程序运行期间，它影响了正常的程序执行流程。

经典的异常成因：

- 循环依赖：a 调用 b，b 调用 a -> 栈溢出异常 `StackOverflowException`
- 除以 0：算术异常 `ArithmeticException: /by 0`

## 异常 `Exception`

**异常**是程序可以自己处理的例外情况，如文件找不到、用户输入错误或网络断联。这些异常一般是由程序逻辑缺陷引起的，程序应该在逻辑上尽可能避免或处理这类异常的发生。

> [!note] 预料并处理异常
> 
> 异常是“可能发生的”例外情况，你无法避免这个可能性，比如网络可能会中断，程序自己没法预料这些异常，遇到了就直接崩溃了，但是你应该预料到它们，并把它们封装为一个个异常，从而在程序中可以用逻辑流程处理它们，让程序能够预料并处理这些异常。

在 Java 中，我们通常会用 `Exception` 类或它的子类来封装程序出现的异常。

### 异常的作用

- 异常用于查询 bug 的关键参考信息。
- 异常可以作为方法的一种特殊返回值，以便通知调用者底层的执行情况。

## 错误 `Error`

**错误**是程序无法自己处理的例外情况，如电脑内存炸了、JVM 自己挂掉了，此时由 JVM 生成并抛出 `Error` 对象。Java 语言开发者会把这些错误封装成各种 `Error` 类，大多数错误与 Java 应用开发者执行的操作无关。Java 应用开发者无法创建 `Error` 的子类。

- **JVM 运行错误 `VirtualMachineError`**：当 JVM 不再有继续执行操作所需的内存资源时，抛 `OutOfMemoryError`。JVM 运行错误发生时，JVM 一般会终止线程。
- JVM 试图执行 Java 应用时发生的错误：这些错误是不可查的，因为它们在应用程序的控制和处理能力之外，而且绝大多数是程序运行时不允许出现的状况，如**类定义错误 `NoClassDefFoundError`** 和**链接错误 `LinkageError`**。

## 三种异常

- **检查性异常 (checked exception)**：最具代表性的检查性异常是用户操作不当问题引起的异常，这是程序员无法预见的。若打开一个不存在的文件，一个异常就发生了，这些异常在编译时不能被简单地忽略。
- **运行时异常 (runtime exception)**：运行时异常是程序员可以避免的异常。与检查性异常相反，运行时异常可以在编译时被忽略。运行时异常是不检查异常，程序可以选择捕获处理，也可以不处理。
- **错误 (error)**：错误不是异常，而是脱离程序员控制的问题，错误在代码中通常被忽略。当栈溢出时，一个错误就发生了，它们在编译时检查不到。

## 异常体系结构

Java 把异常当作对象来处理，并定义一个基类 `java.lang.Throwable` 作为所有异常的超类。Java API 已经定义了许多异常类，它们分为两大类：错误 `Error` 和异常 `Exception`。

![Java 异常体系结构](https://img.xicuodev.top/2026/02/091ad4491346a38db16095eb39423e0f.png "Java 异常体系结构")

- `java.lang.Throwable`
  - `Error`：系统级别错误，属于严重问题。
  - `Exception`：异常，代表程序可能出现的问题。
    - `RuntimeException`：运行时异常，它及其子类是只有在运行时才会暴露的异常，在编译阶段不会出现异常提醒。运行时异常一般是**传递了不合理的参数**导致的异常，如数组索引越界异常。因**用户输入了不合程序预期的值**导致的异常就是典型的运行时异常。
    - 其他异常：编译时异常，编译阶段就会出现异常提醒，目的是提醒程序员**检查本地信息**，如日期解析异常。因**程序员使用了不合编译器预期的值**导致的异常就是典型的编译时异常。

常见的编译时异常：`ParseException` 解析异常、`IOException` 输入输出异常、`SQLException` SQL 查询异常和 `XMLParseException` XML 解析异常。

常见的运行时异常：`ArithmeticException` 算术异常、`NullPointerException` 空指针异常和 `IndexOutOfBoundsException` 索引越界异常 (子类有 `Array` 和 `String` 两种类型的越界异常)。

## 异常处理机制

如果不处理上述三种异常，程序就崩溃了。用异常处理语句处理异常，让程序免于崩溃。

概要：

- 异常处理五个关键字：`try`、`catch`、`finally`、`throw`、`throws`
  - `try {}` 尝试运行代码块，监测异常
  - `catch (Throwable e) {}` 捕获并处理异常
    - `catch` 必须跟 `try` 一起写成 `try-catch` 语句
    - 可以写多个 `catch`，根据实际分几种异常各写一个
    - 父类异常的 `catch` 要写在子类异常的后面
  - `finally {}` 处理善后工作，如关闭 IO 流资源，没有这类工作可以不写
  - `throw`, `throws` 主动抛出异常：一般写在方法中
    - `throw` 写成语句放在方法体中，后跟异常对象：`if (SomeExceptionCond) throw new SomeException();`，`throw` 语句需要放在 `try-catch` 中
    - `throws` 写在方法参数列表后，后跟异常类：`ReturnType someMtd (args) throws SomeException {}`
- IntelliJ IDEA 热键：选中代码按 `Alt+Enter` - 包围 - `try-catch`
- 手动退出程序：`System.exit(1);`

### 交给 JVM 处理

对于异常，Java 虚拟机的默认处理方式是把异常的名称、原因、出现的位置等信息以红色文本打印在控制台，并让程序停止执行，不会再执行异常代码下面的代码。

### 让被调者自行处理

如果让被调者 (callee) 自己处理它逻辑中可能出现的异常，则使用 `try-catch` (`-finally`)。

`try` 中一旦遇到异常，程序会创建一个异常对象并抛出，且不会继续执行 `try` 中异常代码之后的代码。当 `try` 抛出异常时，`catch` 可以捕获并处理该异常对象，并让程序继续往下执行，但不是从 `try` 中断的半途继续，而是执行 `catch` 之后的代码。

需要指定 `catch` 要捕获的异常类型。只有从 `try` 块中抛出了指定类型的异常时，`catch` 分支才会执行。如果 `try` 块可能抛出多种异常，那么需要写多个 `catch`，或者让同一个 `catch` 捕获多种类型的异常，写成 `catch(<异常类型1> | <异常类型2> e)`。如果没有匹配的 `catch`，异常就一层层抛出去了，最终会交给 JVM 处理 (控制台红字报告异常)。

对于存在继承关系的异常类型，父类 `catch` 要写在子类 `catch` 下面。这是由于多态性，父类 `catch` 会把它的所有子类异常一网打尽，这不是我们想要的。为了防止出现我们本能预料但未预料的异常，建议在最下面写 `catch(Exception e)` 捕获异常和它的任何子类来兜底。

无论有没有出现异常，`finally` 分支都必然执行。`finally` 一般用于释放资源，因异常导致的中断不会释放对象占用的资源。

### 交给主调者处理

如果让被调者抛出异常，交给主调者 (caller) 处理，则使用 `throws` 和 `throw` 关键字：

- `throws`：`方法头(参数列表) throws 异常类名1, 异常类名2... {}`
  - 写在方法定义处，表示声明一个异常
  - 告诉调用者，使用本方法可能会有哪些异常
  - 编译时异常：必须要写
  - 运行时异常：可以不写，因为编译器管不到运行时异常，声明了也好，提高可读性

- `throw`：`方法头(参数列表) { throw new NullPointerException(); }`
  - 写在方法内，结束方法
  - 手动抛出异常对象，交给调用者
  - 方法中下面的代码不再执行了

## `Throwable` 的成员方法

- `String getMessage()` 返回此 `throwable` 的详细消息
- `String tostring()` 返回此 `throwable` 的简短描述
- `void printStackTrace()` 把异常的错误信息输出在控制台，只报红字，不会停止程序

> [!tip] 系统标准错误打印流
> 
> `System.err`：是一个 `PrintStream` 字段，用它打印的文本是红色的。

## 自定义异常

使用 Java 内置的异常类可以描述在编程时出现的大部分异常情况，除此之外用户还可以自定义异常，只需继承 `Exception` 类即可。自定义异常可以让控制台的报错信息更加地“顾名思义”。

在程序中使用自定义异常类，大体可分为以下几个步骤：

1. 创建自定义异常类
   1. 写继承关系：运行时异常继承 `RuntimeException`，编译时异常继承 `Exception`
   2. 写空参构造
   3. 写带参构造
   4. 设定报错信息：重写 `toString()` 方法
2. 在可能出现异常的方法中处理异常：使用 `try-catch` 语句捕获并处理
3. 在可能出现异常的方法外处理异常：
   1. 在方法中通过 `throw` 关键字抛出异常对象
   2. 或在方法的声明中通过 `throws` 关键字指明要抛出给方法调用者的异常
   3. 由主调者捕获并处理异常

## 异常实践建议

- 处理运行时异常时，采用逻辑去合理规避同时辅助 `try-catch` 处理
- 在多重 `catch` 块后面，可以加一个 `catch(Exception)` 来处理可能会被遗漏的异常
- 对于不确定的代码，也可以加上 `try-catch`，处理潜在的异常
- 尽量去处理异常，切忌只是简单地调用 `printStackTrace()` 去打印输出
- 具体如何处理异常，要根据不同的业务需求和异常类型去决定
- 尽量添加 `finally` 语句块去释放占用的资源：`IO`、`Scanner` 等
