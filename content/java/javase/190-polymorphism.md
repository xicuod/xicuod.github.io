**多态**是同一方法可以根据调用对象的不同而表现出不同的行为。

- 一个对象的实际类型是确定的，但可以指向对象的引用的类型有很多。
- 方法的多态性允许子类对象隐式向上转型为基类对象，也允许基类对象强制向下转型为子类对象。（把子类换成实现类，基类换成接口，这句话也是成立的。）
- 多态可实现**动态绑定**，方法的实际实现在运行时才知道，使程序的可扩展性和包容性更强。

> [!note] 编译器是假装看不见吗？
> ```java
> Animal a = new Dog();
> a.makeNoise();
> ```
> 在第一行，编译器分明看见了 `Animal` 类型的 `a` 变量引用了一个 `Dog` 类型的对象，为什么到了第二行却说，编译器不知道 `a` 实际引用的对象的类型是什么呢？
>
> 这是因为 Java 是**静态类型语言**，变量的类型在声明时就定死了，永远不会再改变。编译器看到 `a.makeNoise()`，也把 `a` 当作 `Animal` 来看，并且只根据 `Animal` 类里面有没有可用的 `makeNoise()` 方法来判断编译通不通过。编译器当然知道 `new Dog()` 返回的是 `Dog` 类型的对象的引用，但是它也知道 `a` 自始至终都是 `Animal` 类型的变量。至于 `a` 引用的对象的实际类型是什么，这不在编译器的考虑范围内。
>
> 为什么编译器不能聪明一点，根据上下文自动转换变量的类型呢？那是动态类型语言该干的事了，Java 作为静态类型语言，为了类型安全，宁可不要这种伴随着高风险的灵活性。

> [!tip] 我怎么记得类型转换能转换变量的类型？
> 
> 类型转换并不能转换变量的类型。类型转换乍一看像是能把变量的类型都换了，其实这都是编译器和表达式带来的“假象”：
> 
> - `Animal a = new Dog()` 中的 `new Dog()` 表达式返回的引用类型就是 `Dog`，只是因为多态性，`Animal` 类型的 `a` 可以引用 `Dog` 类型的对象，编译器不会抱怨，所谓的隐式转换其实是“对 `Dog` 对象的引用类型从 `Dog` 换成了 `Animal`”。
> - `(Dog) a` 这个表达式返回的引用类型确实是 `Dog`，然而单独的 `a` 的类型也确实仍然是 `Animal`，所谓的显式转换其实是“对 `Dog` 对象的引用类型从 `Animal` 换回了 `Dog`”。

## 多态的前提条件

- 有继承关系
- 子类重写父类方法
- 父类引用指向子类对象（变量和对象有“代差”）

## 多态的好处

多态的核心是[里氏替换原则]()，就是要保证父类引用的行为总是**符合契约的**，不同的子类之间可以安全地互相替换。父类的方法对外约定了什么行为，子类需要照单全收，但它们的内部逻辑是可以多种多样的。父类引用可以 “**透明地**” 使用任何子类重写的方法，把这个方法当作黑盒来用，因为它知道：不论这个方法被谁重写了，都能 “交差”。子类可以放心地执行自己的逻辑，因为它知道：不论自己怎么做事，只要做的是约定的事，就能 “交差”。

这种 “同一接口，多种表现” 的特性，正是多态的价值所在：它让代码能够以统一的方式处理不同类型的对象，同时又保留了各自的行为差异，从而提升代码的灵活性、可扩展性和可维护性。简而言之，多态既保证了 “形式上的统一”(替换的安全性)，又实现了 “内容上的多样”(行为的多变性)。

## 不具有多态性 = 不能重写的成员

- 字段 `field`
- 静态成员 `static`
- 常量成员 `final`
- 私有成员 `private`：首先不能继承
