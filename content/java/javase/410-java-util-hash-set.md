`HashSet` 特点：无序、不重复、无索引

`HashSet` 底层：

- `HashSet` 底层采取[哈希表]()（散列表）存储数据
- 哈希表是一种对于增删改查数据性能都较好的结构
- 哈希表结构组成：
  - JDK 8 之前：数组 + 链表
  - JDK 8 开始：数组 + 链表 + 红黑树

哈希值：

- 根据 `hashCode` 方法算出来的 int 类型的整数，底层数组索引 `int index=(size-1)&哈希值;`
- `hashCode` 方法定义在 `Object` 类中，所有对象都可以调用，默认使用地址值计算
- 一般情况下，会重写 `hashCode` 方法，利用对象内部的属性值计算哈希值
  - 如果没有重写 `hashCode` 方法，相同属性值的不同对象计算出的哈希值是不同的
  - 如果已经重写 `hashCode` 方法，不同对象只要属性值相同，计算出的哈希值就是一样的
  - 哈希碰撞（哈希冲突）：在小部分情况下，不同的属性值或者不同的地址值计算出来的哈希值也有可能一样。
- 如果集合中存储的是自定义对象，必须要重写 `hashCode` 和 `equals` 方法，把地址值换成属性值
- IntelliJ IDEA 快速重写 `hashCode` 和 `equals` 方法：`⌃N` 或 `⌃⏎`

`HashSet` 的底层实现：

1. 创建一个默认长度 16，默认加载因子 (存入 / 总量) 为 0.75 的数组，数组名 `table`
   - 加载因子 0.75：存入数据占比达 0.75 (12/16) 时，数组扩容到原来的两倍 (16 变 32)
2. 根据元素的哈希值跟数组的长度计算出应存入的位置
3. 判断应存位置是否为 `null`，如果是 `null` 直接存入
4. 如果位置不为 `null`，表示有元素，则调用 `equals` 方法比较属性值
5. 一样：不存；不一样：存入数组，形成链表，称为**哈希桶 (hash bucket)**
   - JDK 8 以前：新元素存入数组，老元素挂在新元素下面
   - JDK 8 以后：新元素直接挂在老元素下面，效率更高
6. JDK 8 以后新增：当链表长度大于 8 而且数组长度大于等于 64 时，把过长的链表转为红黑树，提高查找效率

常见问题：

1. `HashSet` 为什么存和取的顺序不一样？
   - 因为 `HashSet` 按哈希值计算底层索引，存的顺序跟底层索引为非线性对应，而取的时候是按底层索引取的
2. `HashSet` 为什么没有索引？
   - 因为 `HashSet` 是数组、链表和红黑树的组合结构，为这种结构设计索引复杂又困难
3. `HashSet` 是利用什么机制保证数据去重的？
   - 用 `hashCode` 方法得到哈希值确定插入位置 (因为存在哈希冲突只相当于粗筛)，用 `equals` 方法比较与哈希桶中对象的属性值是否一致 (细筛)
   - 粗筛和细筛共同保证了 `HashSet` 的无重复性，所以重写时必须同时重写 `hashCode` 和 `equals` 两个方法才能保证正确去重
